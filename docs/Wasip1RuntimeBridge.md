WASIP1 Runtime Bridge Requirements
==================================

This document describes the host-side requirements for implementing a bridge
compatible with [`lib/src/wasm/wit/yara-runtime.wit`](../lib/src/wasm/wit/yara-runtime.wit)
when `yara-x` is built for `wasm32-wasip1`.

The WIT file is the normative ABI. This document explains the behavior that a
working host bridge must provide beyond the bare function signatures.

## Scope

The bridge sits between two different kinds of WebAssembly code:

1. The `yara-x` guest itself, compiled for `wasm32-wasip1`.
2. Core WebAssembly rule modules generated by `yara-x` and passed to the host
   bridge as raw bytes.

The WIT world defines how the guest asks the host to validate, instantiate,
and execute those generated rule modules. The generated rule modules are not
WIT components. They are ordinary core WASM modules.

In other words:

- The bridge ABI is expressed in WIT.
- The modules passed through `validate-module` and `instantiate` are core WASM.
- The bridge must therefore speak both the WIT/component-facing ABI used by
  the guest and the core WASM runtime API used for the generated rule modules.

## High-Level Model

The `wasm32-wasip1` runtime in [`lib/src/wasm/runtime/wasip1.rs`](../lib/src/wasm/runtime/wasip1.rs)
is designed to preserve the same high-level shape as the native Wasmtime-based
runtime:

- `Store` state lives in the guest.
- Actual globals, memories, and instantiated generated modules live in the
  host bridge.
- The guest asks the host bridge to create globals and memories, and later
  refers to them through opaque handles.
- Imported functions in generated rule modules are satisfied by host wrappers
  that call back into the guest through the exported `callbacks` interface.

The host is responsible for managing:

- instance handles
- global handles
- memory handles
- session isolation
- execution timeouts
- guest callback invocation

## Compatibility Rules

The bridge is compatible if, from the guest's point of view, it behaves like a
small Wasmtime-shaped runtime. That implies all of the following:

- Handles returned by the host are opaque `u64` identifiers.
- Every handle is scoped to the `session` that created it.
- A function import wrapper must be able to call back into the guest while the
  generated rule module is executing.
- Export calls must support timeout-driven interruption.
- Memory synchronization must operate on the full linear-memory image, not on
  partial dirty ranges.
- All scalar values crossing the bridge must use the raw bit encodings
  described by the WIT `val-type` enum.

## Session Model

The guest associates each runtime store with a `session_id`. See:

- [`lib/src/scanner/mod.rs`](../lib/src/scanner/mod.rs)
- [`lib/src/scanner/blocks.rs`](../lib/src/scanner/blocks.rs)
- [`lib/src/scanner/context.rs`](../lib/src/scanner/context.rs)
- [`lib/src/wasm/runtime/common.rs`](../lib/src/wasm/runtime/common.rs)

The bridge must treat `session` as part of every resource key.

Required behavior:

- A `global`, `memory`, or `instance` handle is valid only inside the session
  that created it.
- The host must reject cross-session use of handles.
- Callback dispatch must also be session-scoped.
- Session `0` is valid. The guest uses it as the default when no explicit
  runtime session is provided.

Practical requirement:

- Internally, the host should key state by `(session_id, handle)` rather than
  by `handle` alone.

## Opaque Handles

The following values are opaque and must not be interpreted by the host:

- `instance: u64`
- `global: u64`
- `memory: u64`
- `callback-id: u64`

The bridge may choose any handle-allocation strategy, but it must guarantee:

- uniqueness within a session and handle kind while the handle is live
- no accidental aliasing between different live resources
- no reuse that would let a stale handle access a different live resource

The `callback-id` deserves special care:

- It is created by the guest, not by the host.
- In the current implementation it is allocated from a guest-side callback
  registry and remains opaque to the host.
- The host must treat it as an opaque token and pass it back unchanged when
  invoking `callbacks.invoke-callback`.
- The host must not persist callback IDs beyond the lifetime of the imported
  function wrappers that use them.

## Value Encoding

The WIT `val-type` enum describes how raw `u64` transport values are to be
interpreted.

### `val-type`

- `i32`
  - Use the lower 32 bits of the `u64`.
  - When returning an `i32`, zero-extend its bit pattern into `u64`.
- `i64`
  - Use the full 64-bit two's-complement bit pattern.
- `f32-bits`
  - Use the lower 32 bits as the raw IEEE-754 bit pattern of an `f32`.
- `f64-bits`
  - Use the full 64 bits as the raw IEEE-754 bit pattern of an `f64`.

The host must not reinterpret floats numerically and then serialize them
through strings or host-language boxed numbers. The ABI is bit-pattern-based.

### Raw Parameter and Result Lists

Both of these use `list<u64>` with separate type metadata:

- `call-export(..., params, results, ...)`
- `callbacks.invoke-callback(..., args)`

For exported-function calls:

- `params` contains raw arguments.
- `results` tells the host how many results to read and how to decode them.

For imported-function callbacks:

- the host receives the parameter types from `function-import.params`
- the host receives the result types from `function-import.results`
- the host must marshal between the underlying runtime's typed values and the
  guest callback's raw `list<u64>` representation

## WIT Records and Their Meaning

### `function-import`

```wit
record function-import {
  module: string,
  name: string,
  params: list<val-type>,
  results: list<val-type>,
  callback-id: u64,
  sync-flags: u32,
}
```

Required interpretation:

- `module`
  - The import module name expected by the generated core WASM module.
- `name`
  - The exact import field name expected by the generated core WASM module.
- `params`
  - Parameter types for the imported function.
- `results`
  - Result types for the imported function.
- `callback-id`
  - Opaque guest token identifying the callback to invoke.
- `sync-flags`
  - Callback memory-synchronization hints.

Important note about `sync-flags`:

- In the current guest implementation, synchronization semantics are enforced
  in the guest callback trampoline, not in the host wrapper.
- A bridge that simply routes imported-function calls through
  `callbacks.invoke-callback` does not need to interpret `sync-flags` to be
  correct.
- If a host chooses to bypass the guest trampoline logic in some future custom
  implementation, it must preserve the same synchronization semantics.

Current flag meanings from [`lib/src/wasm/runtime/common.rs`](../lib/src/wasm/runtime/common.rs):

- `1 << 0`: synchronize memory before invoking the callback
- `1 << 1`: synchronize memory after invoking the callback

### `extern-import`

```wit
record extern-import {
  module: string,
  name: string,
  value: extern-value,
}
```

Required interpretation:

- `module` and `name` identify the imported item in the generated core WASM
  module.
- `value` points at either a previously created global or memory handle.

The host must bind the generated module's imports exactly as described. It must
not remangle names or infer alternate namespaces.

## Host Functions

### `validate-module`

```wit
validate-module: func(bytes: list<u8>) -> result<_, string>;
```

Purpose:

- Validate the provided core WASM bytes.

Requirements:

- Accept core WASM module bytes emitted by `yara-x`.
- Reject malformed WASM or features unsupported by the host runtime.
- Return `ok` only if the host considers the module valid enough to attempt
  instantiation later.

Recommendations:

- Validation should be side-effect-free.
- Validation does not need to allocate long-lived state.
- The host may validate again during `instantiate`.

What this does not mean:

- This does not validate the WIT bridge itself.
- This does not receive a component binary.

### `instantiate`

```wit
instantiate: func(
  session: u64,
  module-bytes: list<u8>,
  functions: list<function-import>,
  externs: list<extern-import>,
) -> result<u64, string>;
```

Purpose:

- Instantiate the generated core WASM module using the supplied imported
  functions, globals, and memories.

Required behavior:

- Treat `session` as part of instance ownership.
- Validate or revalidate `module-bytes` as needed by the host runtime.
- Resolve imported functions exactly from `functions`.
- Resolve imported globals and memories exactly from `externs`.
- Return a new opaque instance handle.

Each function import wrapper must:

1. Receive typed arguments from the underlying core WASM runtime.
2. Encode them into `list<u64>` according to `function-import.params`.
3. Call the guest export `callbacks.invoke-callback(session, callback-id, args)`.
4. Decode returned raw values according to `function-import.results`.
5. Return those typed results back into the underlying core WASM runtime.

Error behavior:

- If the guest callback returns an error, the wrapper must surface that failure
  to the generated module execution.
- If imports cannot be resolved, return `err(string)`.
- If an `extern-import` refers to an unknown or cross-session handle, return
  `err(string)`.

The host does not need to understand the semantics of imported functions. It
only needs to wire them correctly by `(module, name, params, results)`.

### `instance-destroy`

```wit
instance-destroy: func(session: u64, instance: u64) -> result<_, string>;
```

Purpose:

- Destroy a previously instantiated generated module and free any resources
  owned by that instance.

Requirements:

- Destroy only the instance belonging to `session`.
- Reject unknown or cross-session instance handles.
- Release runtime resources owned by that instance.

Practical note:

- The guest currently ignores errors returned from `instance-destroy` during
  `Drop`, but a correct bridge should still clean up accurately and return
  meaningful errors for debugging.

### `call-export`

```wit
call-export: func(
  session: u64,
  instance: u64,
  name: string,
  params: list<u64>,
  results: list<val-type>,
  timeout-nanos: u64,
) -> result<list<u64>, string>;
```

Purpose:

- Invoke an exported function from a generated core WASM instance.

Required behavior:

- Look up the instance by `(session, instance)`.
- Find the export named `name`.
- Marshal `params` into the runtime's typed calling convention.
- Invoke the export.
- Return raw `u64` results matching `results`.

Timeout requirements:

- `timeout-nanos == u64::MAX` means "no timeout".
- Any other value is a remaining budget from the guest's wall-clock deadline.
- If the export call exceeds that budget, return:

```text
__yarax_timeout__
```

The exact string matters. The guest treats that error string as the timeout
sentinel and converts it into `ScanError::Timeout`.

If the host times out the call in some other way and returns a different error
string, the guest will treat it as a generic runtime failure instead of a scan
timeout.

Current usage:

- Today the guest calls only the `main` export with:
  - `params = []`
  - `results = [i32]`

Full compatibility requirement:

- The bridge should still honor the general WIT contract for arbitrary
  `params` and `results`, even if current `yara-x` only exercises a subset.

### `global-new`

```wit
global-new: func(
  session: u64,
  val-type: val-type,
  mutable: bool,
  value: u64,
) -> result<u64, string>;
```

Purpose:

- Create a host-side global and return an opaque handle.

Requirements:

- Associate the new global with `session`.
- Interpret `value` according to `val-type`.
- Respect `mutable`.
- Return a new opaque global handle.

### `global-get`

```wit
global-get: func(
  session: u64,
  global: u64,
  val-type: val-type,
) -> result<u64, string>;
```

Purpose:

- Read the current value of a host-side global.

Requirements:

- Look up the global by `(session, global)`.
- Read its current value.
- Return that value encoded according to `val-type`.

Recommendation:

- Validate that the requested `val-type` matches the stored global type.

Current guest quirk:

- If `global-get` returns an error, the guest currently substitutes the zero
  value for the requested type.
- A bridge should not rely on that behavior. In normal operation,
  `global-get` should succeed for valid handles.

### `global-set`

```wit
global-set: func(
  session: u64,
  global: u64,
  val-type: val-type,
  value: u64,
) -> result<_, string>;
```

Purpose:

- Write a new value into a host-side global.

Requirements:

- Look up the global by `(session, global)`.
- Reject writes to immutable globals.
- Interpret `value` according to `val-type`.
- Update the global.

### `memory-new`

```wit
memory-new: func(
  session: u64,
  initial: u32,
  maximum: option<u32>,
) -> result<u64, string>;
```

Purpose:

- Create a host-side linear memory and return an opaque handle.

Requirements:

- Memory page size is 64 KiB, matching core WASM semantics.
- Associate the memory with `session`.
- Allocate at least `initial` pages.
- Honor `maximum` if the underlying runtime supports it.

Current `yara-x` behavior:

- The guest creates the main memory with `maximum == Some(initial)`.
- That means fixed-size memory is sufficient for compatibility today.

### `memory-read`

```wit
memory-read: func(session: u64, memory: u64) -> result<list<u8>, string>;
```

Purpose:

- Return the current contents of the full linear memory.

Requirements:

- Look up the memory by `(session, memory)`.
- Return the full byte contents from offset `0` to the current memory length.

This is a full snapshot operation. The current guest runtime does not request
partial reads.

### `memory-write`

```wit
memory-write: func(
  session: u64,
  memory: u64,
  data: list<u8>,
) -> result<_, string>;
```

Purpose:

- Replace the current memory contents with `data`.

Requirements:

- Look up the memory by `(session, memory)`.
- Write the provided bytes starting at offset `0`.
- Preserve the exact byte contents supplied by the guest.

Compatibility expectation:

- The guest currently uses full-memory synchronization in both directions.
- Bridges should therefore treat `memory-write` as a full-image write, not as
  a sparse patch.
- Because the current guest allocates fixed-size memory, the safest behavior
  is to reject writes whose length does not match the current memory size.

## Guest Callback Export

The host must be able to invoke the guest-side callback export described by:

```wit
interface callbacks {
  invoke-callback: func(
    session: u64,
    callback-id: u64,
    args: list<u64>,
  ) -> result<list<u64>, string>;
}
```

Purpose:

- Allow imported functions in generated rule modules to call back into guest
  code.

Required behavior:

- Pass the original `session`.
- Pass the `callback-id` exactly as supplied in `function-import`.
- Pass raw `args` encoded according to the function's parameter types.
- Receive raw results encoded according to the function's result types.

Guest-side validation:

- The guest rejects callbacks whose `session` does not match the session that
  created the callback.
- The guest rejects unknown callback IDs.

### Calling Through Generated Bindings

If the host uses generated WIT/component bindings for the `runtime` world,
callback invocation and post-return cleanup should be handled automatically by
those bindings.

### Calling Through Raw Export Symbols

If the host does not use generated bindings and instead calls the guest export
symbols directly, it must match the canonical ABI used by `wit-bindgen`.

In the current guest, the relevant exports are:

- `yara:runtime/callbacks#invoke-callback`
- `cabi_post_yara:runtime/callbacks#invoke-callback`

Important requirement:

- After reading the callback result, the host must call the matching
  `cabi_post_...` function to release guest-allocated return buffers.

If the host skips the post-return call, callback invocation will leak memory in
the guest.

## Re-Entrancy Requirements

The bridge must tolerate this control flow:

1. The guest asks the host to `call-export` on a generated rule module.
2. The generated rule module calls an imported function.
3. The host wrapper for that import calls back into the guest through
   `callbacks.invoke-callback`.
4. The guest callback may itself read or write synchronized memory state.
5. Control returns to the generated rule module.
6. Control eventually returns to the guest from `call-export`.

That means the bridge must support nested guest-host-guest transitions without:

- losing track of the active session
- corrupting instance state
- corrupting memory state
- deadlocking its own runtime

## Memory Synchronization Semantics

The current guest runtime keeps a local byte-vector cache for each host memory.

Important implications:

- Before certain callbacks, the guest may pull the latest host memory with
  `memory-read`.
- After certain callbacks, the guest may push its cached memory back to the
  host with `memory-write`.
- Before `call-export`, the guest writes its current cached memory to the host.
- After `call-export`, the guest reads the full memory back from the host.

This means a compatible bridge must make memory operations strongly coherent at
function-call boundaries. The host must not delay or batch these operations in
a way that changes what the guest observes.

## Error Handling

All host functions return `result<_, string>`.

Requirements:

- Use `ok(...)` only for successful completion.
- Use `err(string)` for all operational failures.
- Error strings should be human-readable because they may surface in runtime
  failures.

Special rule:

- `call-export` must use the exact error string `__yarax_timeout__` to report
  a timeout.

Suggested categories of host error:

- invalid or unsupported module bytes
- unknown handle
- cross-session handle use
- immutable-global write attempt
- memory size mismatch
- missing export
- underlying runtime trap or instantiation error

## Minimal Bridge Implementation Strategy

A minimal compatible host bridge can be implemented with the following design:

1. Maintain a per-session table of:
   - globals
   - memories
   - instances
2. Implement `validate-module` with the host's core WASM validator.
3. Implement `global-new/get/set` and `memory-new/read/write` as thin wrappers
   over the host runtime's resource objects.
4. During `instantiate`:
   - create host function wrappers from `function-import`
   - resolve `extern-import` handles
   - instantiate the generated core WASM module
   - store the instance under a new handle
5. During `call-export`:
   - resolve the instance
   - enforce the timeout budget
   - invoke the named export
   - return raw result words
6. During callback invocation:
   - call the guest `callbacks.invoke-callback`
   - always perform post-return cleanup if not using generated bindings

## Compatibility Checklist

Before considering the bridge complete, verify all of the following:

- `validate-module` accepts valid `yara-x` generated core WASM modules.
- `instantiate` can wire both function imports and extern imports.
- `call-export` can invoke `main` and return one `i32` result.
- timeout expiration from `call-export` returns `__yarax_timeout__`.
- globals are isolated by session.
- memories are isolated by session.
- instances are isolated by session.
- imported functions can call back into the guest.
- callback IDs are treated as opaque tokens.
- raw scalar values are encoded exactly as specified by `val-type`.
- `memory-read` returns the full memory image.
- `memory-write` writes the full memory image.
- raw callback invocation performs post-return cleanup if bindings are not used.
